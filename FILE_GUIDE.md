# 📂 File Guide — Architecture 3.0

> What every file and folder does, and how they relate to each other.

---

## Root `.md` Files

These five documents each serve a different reader at a different moment. They are not redundant — each has a unique job.

---

### `MASTER_GUIDE.md` — The front door

**Reader:** Anyone, first thing.  
**When:** The very first time you open this project.  
**Job:** Orient you in 60 seconds. Explains the pipeline at a glance, points you to the right doc for your role (human or agent), and lists the project structure. Does not go deep on anything — its only goal is to tell you where to go next.

---

### `HOW_TO_MAKE_A_COMIC.md` — The human operator's manual

**Reader:** The human sitting at the keyboard.  
**When:** Every time you run a new phase.  
**Job:** Step-by-step instructions for the human's side of the pipeline. Explains what to do, what to look for in the app, how to flag issues, and what "approval" means at each phase. Written assuming the human knows nothing about the technical internals.

---

### `AGENT_GUIDE.md` — The technical contract for AI agents

**Reader:** AI agents only.  
**When:** At the start of every agent session.  
**Job:** The authoritative technical reference. Covers exact file paths, JSON schemas, API endpoints, QA tag definitions, and agent behavior rules. An agent dropped cold into this project should be able to fully orient from this doc alone — without reading anything else.

---

### `AGENT_HANDOFF.md` — The session snapshot

**Reader:** The next AI agent picking up the project.  
**When:** At the start of a new agent session.  
**Job:** Unlike the other docs, this is a *living* file — it gets updated as the project progresses. It records the current state: which data files exist, what phase is next, what tasks are pending. Think of it as the "where we stopped" note between sessions.

---

### `init_project_protocol.md` — The one-time setup script

**Reader:** AI agent, at project creation only.  
**When:** Once, before Phase 0, when starting a brand-new comic from this blank template.  
**Job:** Instructs the agent how to customize the generic `pipelines/` files for a specific project (fill in scene types, tone rules, structural tags). After setup is complete, this doc is not used again for that project.

---

### `PRODUCTION_STATUS.md` — The Kanban board

**Reader:** Both human and agents.  
**When:** Continuously updated throughout production.  
**Job:** Tracks the status of every phase with tags: `[TODO]`, `[IN PROGRESS]`, `[REVIEW]`, `[APPROVED]`. An `[APPROVED]` phase is locked — neither human nor agent should modify its output without explicit instruction.

---

### `FILE_GUIDE.md` — This document

**Reader:** Anyone who needs a map.  
**When:** Whenever you're confused about what a file or folder does.  
**Job:** Explains the role of every file and folder in the project root. Update it when new files are added.

---

## Folders

---

### `data/` — All live JSON outputs

The single source of truth for all generated content. Every phase reads from here and writes back here. Files are flat (no subfolders except `images/`).

| File | Generated by | Contains |
|------|-------------|---------|
| `lore.json` | Phase 0 | World rules, tone, genre, visual style, character sheets |
| `scenario.json` | Phase 0 | Full scene list with emotional beats and anecdotes |
| `pages.json` | Phase 1 | Page-by-page breakdown with type and dramatic focus |
| `intro_pages.json` | Phase 1.5 | Character intro page layouts with panels and dialogue |
| `panels.json` | Phase 2 | Per-panel framing, action, characters, structural tags |
| `script.json` | Phase 3 | All dialogue and captions with permanent IDs |
| `images/page_N/panel_N.png` | Phase 4/5 | Generated panel artwork |

> ⚠️ Never manually delete or rename files in `data/`. Downstream phases reference them by exact path.

---

### `qa/` — QA reports, organized by phase

Contains markdown reports exported from the Comic Studio app when the human flags issues. Each report is timestamped and never overwritten.

| Subfolder | Phase |
|-----------|-------|
| `qa/lore/` | Phase 0 |
| `qa/pacing/` | Phase 1 |
| `qa/characters/` | Phase 1.5 |
| `qa/structure/` | Phase 2 |
| `qa/script/` | Phase 3 |
| `qa/images/` | Phase 4/5 |

**How agents use these:** Before regenerating any output, scan the relevant `qa/[phase]/` folder, sort by timestamp (oldest first), and apply each flagged change surgically. Append `[APPLIED]` to each report after processing.

---

### `pipelines/` — Agent instruction files

These are the *active working instructions* for each phase. They are customized at project start (via `init_project_protocol.md`) to fit the specific project's tone, scene types, and structural rules.

**Phase 0 — Pre-production (3 files run in sequence):**
| File | Job |
|------|-----|
| `01_style_building.md` | Defines world type, tone, visual style |
| `02_character_distillation.md` | Builds character sheets |
| `03_scenario_development.md` | Produces the full scene list |

**Also in `pipelines/`:**
| File | Phase | Job |
|------|-------|-----|
| `04_visual_signatures.md` | Phase 0 | Visual identity and signature details per character |
| `pacing_instructions.md` | Phase 1 | Rules for distributing scenes across pages |
| `intro_instructions.md` | Phase 1.5 | Rules for character intro page structure |
| `structuring_instructions.md` | Phase 2 | Rules for panel breakdown and structural tags |
| `scripting_instructions.md` | Phase 3 | Tone rules, dialogue style, scene-type guidelines |
| `generation_instructions.md` | Phase 4/5 | Prompt structure, visual style baseline, fix procedures |

> ⚠️ These files contain project-specific content filled in during initialization. Do not treat them as generic templates once a project is running.

---

### `docs/` — Per-phase agent reference docs

Deeper reference material for each phase. Where `pipelines/` files tell the agent *what* to produce, `docs/` files explain *how* — context, edge cases, schema details, and common pitfalls.

| File | Phase |
|------|-------|
| `phase_0_lore.md` | Phase 0 |
| `phase_1_pacing.md` | Phase 1 |
| `phase_1b_characters.md` | Phase 1.5 |
| `phase_2_structure.md` | Phase 2 |
| `phase_3_script.md` | Phase 3 |
| `phase_4_images.md` | Phase 4/5 |
| `phase_6_assembly.md` | Phase 6 |

**Rule:** Read the relevant `docs/phase_N_*.md` before executing any phase.

---

### `app/` — Comic Studio 3.0 (the React dashboard)

The browser-based review tool. **Does not run agents.** It is purely for human review, inline editing, and QA report export.

**To start it:**
```
cd app
npm run dev
# → http://localhost:5173/
```

**What each tab does:**

| Tab | Phase | What you can do |
|-----|-------|----------------|
| 🌍 Lore & Story | 0 | Read lore and scenario; flag fields for agent |
| 📋 Pacing | 1 | Review page breakdown; flag pages for changes |
| 🎭 Characters | 1.5 | Review intro pages; flag for rewrites |
| 📐 Panel Structure | 2 | Review panels; change framing inline (saves instantly); flag for agent |
| ✍️ Script | 3 | Review dialogue; edit text inline (saves instantly); flag for agent |
| 🧩 Assembly | 6 | Manual canvas: drag panels, place speech bubbles, adjust properties |

**API (served by the Vite dev plugin — agents can call these directly):**
- `GET /api/load?path=data%2Fscript.json` — reads any file relative to the project root
- `POST /api/save` — `{ path: "data/panels.json", content: {...} }` — writes JSON
- `POST /api/save-qa` — `{ path: "qa/script/qa_report_...", content: "markdown" }` — writes a QA report

All paths are relative to the `architecture 3.0/` project root (one level above `app/`).
